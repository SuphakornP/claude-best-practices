🚀 ติดตั้ง & เริ่มต้นใช้งาน

ความต้องการ: macOS 13+ / Linux (Ubuntu 20.04+) / Windows 10+, RAM 4GB+, Node.js 18+ (เฉพาะถ้าติดตั้งผ่าน npm) และต้องมี paid plan (Pro / Max / Team / Enterprise) — free plan ใช้ไม่ได้

install.sh — ติดตั้ง Claude Code

curl -fsSL https://claude.ai/install.sh | bash        # macOS / Linux
irm https://claude.ai/install.ps1 | iex               # Windows PowerShell

# วิธีที่ 2: npm (เมื่อต้องการ pin version)
npm install -g @anthropic-ai/claude-code
# ⚠️ ห้ามใช้ sudo npm install -g เด็ดขาด (จะเกิดปัญหา permission)

# ตรวจสอบ + เริ่ม session แรก
claude --version
claude doctor          # วินิจฉัยปัญหา environment
cd your-project && claude   # ครั้งแรกจะเปิด browser ให้ login

🎛️ เลือก Model & Effort — จับคู่ "เกียร์" ให้ถูกงาน

มี 2 คันโยกที่ควบคุมคุณภาพ-ความเร็ว-การกิน usage limit: Model (ใช้สมองรุ่นไหน) × Effort (ให้คิดลึกแค่ไหนต่อรอบ) — เอกสาร Anthropic ชี้ว่า "การปรับ effort มักเป็นคันโยกที่ดีกว่าการสลับ model" เพราะได้ trade-off ละเอียดกว่าในรุ่นเดียวกัน

🧠 คันโยกที่ 1: เลือก Model family

Modelจุดแข็งใช้กับงาน
Haiku 4.5 🐇เร็วสุด กิน limit น้อยสุดrename / format / lookup ง่ายๆ / fan-out script จำนวนมาก
Sonnet 4.6 ⚖️สมดุลที่สุด — daily driverfeature ปกติ, bugfix ทั่วไป (default ของ Pro / Team Standard)
Opus 4.x 🧠reasoning ลึกarchitecture, debugging ข้ามหลายไฟล์, refactor ใหญ่, code review เชิงลึก (default ของ Max / Team Premium)
Fable 5 ✨tier เหนือ Opus — investigate ก่อนลงมือ + verify งานตัวเองบ่อยกว่างานยาวเกินหนึ่งนั่ง / root-cause คลุมเครือ — ไม่เป็น default ต้องเลือกเองด้วย /model fable แล้ว "บอก outcome ไม่ใช่ steps"
  • สลับ model กลาง session ได้โดย context ไม่หาย — เปิดงานด้วย Sonnet แล้วค่อยยกระดับเป็น Opus เฉพาะตอนติดหิน
  • opusplan = โหมด hybrid: plan ด้วย Opus, execute ด้วย Sonnet — ได้แผนระดับ Opus ในราคาการลงมือระดับ Sonnet
  • session ยาว / โค้ดเบสใหญ่: sonnet[1m] / opus[1m] = context 1 ล้าน token · alias best = Fable 5 ถ้า org มีสิทธิ์ ไม่งั้น Opus ล่าสุด
  • บนแผน subscription เมื่อใช้ Opus ใกล้โควตา ระบบอาจสลับลง Sonnet ให้อัตโนมัติ — เช็ค model ปัจจุบันด้วย /status (ระดับ effort โชว์ข้าง spinner ตลอดเวลา)
  • Fable 5 มี safety classifier ในตัว: งานสาย cybersecurity / biology จะถูก re-route ไปรันบน Opus อัตโนมัติ — เป็นพฤติกรรมปกติของระบบ ไม่ใช่บัญชีโดน flag

⚡ คันโยกที่ 2: Effort ladder (ตั้งผ่าน /effort)

1

🌱 low

ตอบไวจาก pattern แทบไม่คิด — งาน routine, ยืด limit ได้ไกลสุด

2

🚶 medium

คิดพอประมาณ — งาน coding ประจำวันส่วนใหญ่

3

🏃 high ★

สมดุลคุณภาพ/ความเร็วดีสุด — default ของ Fable 5, Opus 4.8/4.6, Sonnet 4.6

4

🔥 xhigh

จุดที่ docs บอกว่า "ดีสุดสำหรับงาน coding/agentic ส่วนใหญ่" บน Opus 4.7/4.8 (default ของ 4.7)

5

🌋 max

คิดลึกสุด ไม่จำกัด token — เฉพาะ session นั้น ใช้กับปัญหาหินสุดเท่านั้น

↑ low–xhigh จำค่าข้าม session · max ใช้ครั้งเดียวจบ · ถ้าตั้งระดับที่รุ่นไม่รองรับ ระบบ fallback ลงระดับสูงสุดที่รองรับให้เอง (เช่น xhigh → high บน Opus 4.6) · ในเมนูยังมี ultracode = ส่ง xhigh + ให้ Claude Code จัด workflow แบบ dynamic เพิ่ม

💭 แล้ว "ultrathink" ล่ะ?

รุ่นตั้งแต่ Opus 4.6 / Sonnet 4.6 ขึ้นไปใช้ adaptive reasoning — โมเดลตัดสินเองว่า step ไหนควรคิดลึกแค่ไหน โดยมี effort เป็นคันโยกหลัก (Opus 4.7+ และ Fable 5 เป็น adaptive เสมอ ปิดไม่ได้) ส่วน keyword พิเศษเหลือคำเดียวคือ ultrathink: พิมพ์ไว้ตรงไหนก็ได้ใน prompt เพื่อขอให้คิดลึกขึ้นเฉพาะ turn นั้น โดยไม่เปลี่ยน effort ของ session — ⚠️ ระวังความเชื่อเก่า: think / think hard / think harder ตอนนี้เป็นแค่ข้อความธรรมดา ไม่ใช่ keyword แล้ว

⚠️ Effort สูง ≠ ดีเสมอ: งานง่ายที่โดนสั่งให้คิดหนักจะ overthink — ช้าลง เปลือง limit และบางครั้งคำตอบแย่ลงด้วยซ้ำ กฎจำง่าย: "ถ้ามนุษย์คิดมากเกินไปแล้วทำงานนั้นแย่ลง Claude ก็เช่นกัน" เริ่มที่ default แล้วปรับขึ้นเฉพาะเมื่อเจองานที่คำตอบตื้นไป
🧑‍💼 ฝั่ง Cowork / Claude app ก็เลือกได้: กดชื่อ model ข้างช่องพิมพ์ → เลือกรุ่น (Haiku / Sonnet / Opus / Fable) และปรับ Effort ได้จาก picker เดียวกัน — ค่า default คือสมดุลที่เหมาะกับงานส่วนใหญ่ ลดระดับเมื่ออยากได้คำตอบไว/ประหยัด limit และเก็บระดับสูงสุดไว้กับโจทย์ยากจริงๆ · จำไว้ว่า Haiku/Sonnet "ประหยัดโควตา" กว่า Opus/Fable ชัดเจน — ถ้างานนั้น Sonnet เอาอยู่ การใช้ Opus คือจ่าย limit ฟรี · ฝั่งองค์กร admin ตั้ง default model ให้ทั้งทีมได้

🔐 Permissions & Safety

โดย default Claude Code ขออนุญาตก่อนทำสิ่งที่เปลี่ยนแปลงระบบ (เขียนไฟล์, รัน bash, เรียก MCP tools) ลดการกดยืนยันซ้ำๆ ได้ 3 ทาง:

  • Auto mode — ให้ classifier ตรวจคำสั่ง บล็อกเฉพาะที่เสี่ยง: claude --permission-mode auto
  • Allowlist — อนุญาตคำสั่งที่รู้ว่าปลอดภัยถาวร เช่น npm run lint, git commit ผ่าน /permissions
  • Sandbox — /sandbox แยกระดับ OS จำกัด filesystem/network สำหรับงานที่ไม่ไว้ใจ
.claude/settings.json — deny rules ป้องกัน secrets
{
  "permissions": {
    "allow": ["Bash(npm run lint)", "Bash(npm run test:*)", "Bash(git commit *)"],
    "deny":  ["Read(./.env)", "Read(./.env.*)", "Read(./secrets/**)"]
  }
}
⛔ ห้ามติดนิสัย: --dangerously-skip-permissions ใช้เฉพาะใน sandbox/container ที่ scope แคบจริงๆ เท่านั้น (เคยมี CVE ที่ deny rule ถูก bypass ได้ — อัปเดต Claude Code เป็นเวอร์ชันล่าสุดเสมอ)

🔌 MCP Servers — ต่อ Claude เข้ากับโลกภายนอก

MCP (Model Context Protocol) เป็น open standard ของ Anthropic (ยอดดาวน์โหลด ~100M/เดือน กลายเป็นมาตรฐานอุตสาหกรรม) ใช้เชื่อม Claude กับ tools / databases / APIs

mcp — คำสั่งที่ใช้บ่อย
# เพิ่ม remote HTTP server
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp

# เพิ่ม local stdio server พร้อม env var
claude mcp add --env AIRTABLE_API_KEY=YOUR_KEY \
  --transport stdio airtable -- npx -y airtable-mcp-server

claude mcp list        # ดู servers ทั้งหมด + สถานะ
claude mcp get sentry  # ตรวจรายตัว
# ใน session: พิมพ์ /mcp เพื่อ auth OAuth + ดูสถานะ
Scopeเก็บที่เหมาะกับ
local~/.claude.jsonใช้ส่วนตัว เฉพาะ project นี้ (default)
project.mcp.json (commit ได้)แชร์ให้ทั้งทีมใช้เหมือนกัน ✅
userconfig ระดับเครื่องใช้ทุก project ของคุณ

🔥 ที่นิยม: GitHub Playwright Sentry PostgreSQL Figma Context7 — ติดตั้งจากแหล่งที่เชื่อถือได้เท่านั้น

⚡ Power Tools: Skills · Slash Commands · Hooks · Subagents

4 กลไกขยายความสามารถ — เลือก tab เพื่อดูว่าแต่ละตัวต่างกันยังไงพร้อมไฟล์ตัวอย่าง

Skills = ความรู้ที่ Claude หยิบใช้เอง เมื่องานตรงกับ description — เป็น folder ที่มีได้ทั้ง instructions, scripts และ assets ปัจจุบัน custom slash commands ถูก merge เข้ากับระบบ Skills แล้ว (ถ้าชื่อชนกัน skill ชนะ) จึงแนะนำให้เริ่มที่ .claude/skills/<name>/SKILL.md

.claude/skills/release-notes/SKILL.md
---
name: release-notes
description: สร้าง release notes จาก git log ระหว่างสอง tag ใช้เมื่อผู้ใช้ขอ changelog หรือ release notes
---
# Release Notes Generator
1. รัน `git log --oneline {{prev_tag}}..HEAD`
2. จัดกลุ่ม commit เป็น Features / Fixes / Breaking Changes
3. เขียนภาษาคนอ่านรู้เรื่อง ไม่ copy commit message ดิบ
4. บันทึกเป็น RELEASE_NOTES.md

Slash commands = ทางลัดที่คุณเรียกเอง — ไฟล์ markdown ใน .claude/commands/ (ชื่อไฟล์ = ชื่อคำสั่ง) รับค่าผ่าน $ARGUMENTS หรือ $1 $2 และตั้ง frontmatter จำกัด tools ได้

.claude/commands/security-scan.md → ใช้: /security-scan
---
allowed-tools: Read, Grep, Glob
description: สแกนหา security vulnerabilities ใน codebase
---
Analyze the codebase for security vulnerabilities including:
- SQL injection risks
- XSS vulnerabilities
- Exposed credentials / secrets in code
Report findings with file paths + line numbers. Focus: $ARGUMENTS

Hooks = กฎที่รันแน่นอน 100% (deterministic) ต่างจาก CLAUDE.md ที่เป็นแค่คำแนะนำ — เหมาะกับงานที่ "ต้องเกิดเสมอ" เช่น auto-format หลังแก้ไฟล์ทุกครั้ง

.claude/settings.json — hook: lint ทุกครั้งหลัง edit
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command",
                  "command": "npx eslint --fix \"$CLAUDE_FILE_PATHS\"" }]
    }]
  }
}

Subagents = ผู้ช่วยที่มี context แยกของตัวเอง — เหมาะกับงานอ่านไฟล์เยอะหรือ review เพราะไม่กิน context หลัก เรียกตรงๆ ได้: "use a subagent to review this code for security issues"

.claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication / authorization flaws
- Secrets or credentials in code
Provide specific line references and suggested fixes.

🌳 Git Worktrees: รันหลาย Claude พร้อมกัน

💬 Prompting เฉพาะทาง: Before → After

กดสลับ ❌/✅ เพื่อเทียบ — ความต่างเดียวที่สำคัญที่สุดคือ ความเฉพาะเจาะจง + บอกวิธี verify

1 · Scope งานให้ชัด
add tests for foo.py
write a test for foo.py covering the edge case where the user is logged out. avoid mocks.
💡 ระบุ edge case ที่ต้องครอบคลุม + ข้อห้าม (no mocks) → ได้ test ที่มีความหมายจริง
2 · ชี้แหล่งข้อมูลให้ค้น
why does ExecutionFactory have such a weird api?
look through ExecutionFactory's git history and summarize how its api came to be
💡 บอก "ค้นจากไหน" (git history) → คำตอบอิงหลักฐาน ไม่ใช่เดา
3 · อ้าง pattern ที่มีอยู่แล้วใน codebase
add a calendar widget
look at how existing widgets are implemented on the home page. HotDogWidget.php is a good example. follow that pattern to add a calendar widget.
💡 ให้ตัวอย่างจริงในโปรเจกต์ → โค้ดใหม่กลืนกับของเดิม ไม่สร้างสไตล์แปลกปลอม
4 · เล่าอาการ + จุดเริ่มสืบ ไม่ใช่สั่ง "แก้บั๊ก"
fix the login bug
users report login fails after session timeout. check the auth flow in src/auth/, write a failing test that reproduces it, then fix.
💡 อาการ → ตำแหน่งที่น่าสงสัย → reproduce ด้วย test ก่อนแก้ = bug fix ที่พิสูจน์ได้
5 · ให้เกณฑ์ verify เสมอ
implement a function that validates email
write validateEmail(). test cases: "user@example.com" → true, "invalid" → false. run the tests after implementing.
💡 มี input/output คาดหวัง + สั่งรัน test → Claude ตรวจงานตัวเองได้ ("ถ้า verify ไม่ได้ อย่า ship")

⚠️ Pitfalls ที่เจอบ่อย (คลิกเพื่อดูวิธีแก้)

🍲 Kitchen-sink session — ยัดหลายงานใน context เดียว
อาการ: เริ่มจากแก้บั๊ก ลามไป refactor ลามไปถามเรื่องอื่น คำตอบเริ่มเพี้ยนเพราะ context ปนกันหมด
วิธีแก้: หนึ่ง session = หนึ่งงาน — /clear ทุกครั้งที่ขึ้นงานใหม่
🔁 แก้วนซ้ำกับ approach ที่พังแล้ว
อาการ: บอกให้แก้ → ยังผิด → บอกอีก → ผิดแบบใหม่ เพราะ context เต็มไปด้วยความพยายามที่ fail
วิธีแก้: กฎ 2 ครั้ง — /clear แล้วเขียน prompt ใหม่ที่สรุปบทเรียน เช่น "อย่าใช้วิธี X เพราะติด Y"
📜 CLAUDE.md ยาวจน Claude เริ่มเมินกฎ
อาการ: เขียนกฎไว้แต่ Claude ทำผิดอยู่ดี — มักเกิดเมื่อไฟล์บวมเกิน ~300 บรรทัด
วิธีแก้: ตัดด้วยกฎทอง "ลบแล้วพังไหม?" + ย้ายกฎที่ต้องเกิด 100% ไปเป็น hook แทน
🤝 Trust-then-verify gap — โค้ดดูดีแต่ไม่ handle edge case
อาการ: รับโค้ดที่ "อ่านแล้วใช่" โดยไม่รัน — แตกตอน production
วิธีแก้: ทุกงานต้องมีกลไก verify (test / screenshot / รันจริง) — ถ้า verify ไม่ได้ อย่า ship
🏗️ Over-engineering — abstraction เกินจำเป็น
อาการ: ขอ function เดียว ได้ factory + interface + 3 ไฟล์ใหม่
วิธีแก้: ใส่ใน CLAUDE.md: use the simplest possible approach; no new abstractions unless asked
🌀 Infinite exploration — สั่ง "investigate" ไม่จำกัดขอบเขต
อาการ: Claude ไล่อ่านครึ่ง repo เผา context หมดก่อนได้คำตอบ
วิธีแก้: scope ให้แคบ ("ดูเฉพาะ src/auth/") หรือโยนงานสำรวจให้ subagent เพื่อรักษา context หลัก

🛠️ Skills — เอกสารระดับมืออาชีพแบบ "read before create"

🔒 Safety — สิ่งที่ทุกคนในทีมต้องรู้ก่อนใช้

  • Isolated VM: โค้ด/shell ทั้งหมดรันในกล่องแยก ไม่แตะ OS หลักโดยตรง
  • Controlled access: อ่าน/เขียนได้เฉพาะ folder ที่คุณอนุญาต, network ตามที่ตั้งค่า
  • Deletion protection: ขออนุญาตก่อนลบไฟล์ถาวรเสมอ ทุกโหมด

🎣 Prompt Injection — ภัยอันดับหนึ่งที่ต้องสอนทุกคน

🛡️ แนวปฏิบัติ: ① ใช้ "Ask before acting" กับงาน/แหล่งข้อมูลใหม่เสมอ ② จำกัดการท่องเว็บให้อยู่ในไซต์ที่เชื่อถือได้ ③ ใช้ MCP / plugin / skill จากแหล่ง official เท่านั้น ④ ระวังเป็นพิเศษกับ computer use — ไม่มี sandbox ระหว่าง Claude กับหน้าจอจริง การคลิกลิงก์ใน app หนึ่งอาจพาไปเปิด app อื่นที่ไม่ได้ตั้งใจอนุญาต

🎓 หลัก Prompting 6 ข้อ ใช้ได้ทั้งสองฝั่ง

  • 1. ชัดเจน ตรง ละเอียด — คิดว่า Claude คือ intern เก่งมากแต่เพิ่งมาวันแรก: ให้บริบท เป้าหมาย และข้อจำกัดครบ
  • 2. ให้ตัวอย่าง (few-shot) — แนบ 2–5 ตัวอย่าง input/output ที่ดี ครอบด้วย tag อย่าง <example> เพิ่มความแม่นยำชัดเจน
  • 3. ใช้ XML tags แยกส่วน — <instructions> <context> <data> ช่วยให้ไม่ปนกัน (Claude ถูก fine-tune ให้สนใจ tags เป็นพิเศษ)
  • 4. ให้คิดทีละขั้น — งานที่ต้อง reasoning ให้บอก "think step by step ก่อนตอบ"
  • 5. ระบุ output format — JSON / ตาราง / ความยาว / ภาษา อย่าให้เดา
  • 6. ทดสอบและ iterate — prompt ที่ดีเกิดจากการลองและปรับ ไม่ใช่เขียนครั้งเดียวเป๊ะ
🥇 Golden rule: เอา prompt ของคุณให้เพื่อนร่วมงานที่ไม่รู้บริบทอ่าน — ถ้าเขางง Claude ก็งง
